# 写作习惯与风格手册

> 性质：个人写作方法论的持续积累，供写作前参考，写作后补充
> 更新方式：与 AI 讨论过程中逐渐提炼，发现新的有效模式时追加
> 关联文件：`AI思维碎片/` 记录思想内容，本文档记录表达方式

---

## 一、核心写作哲学

### 1.1 换位思考：从读者的决策链倒推文章结构

这是所有写作习惯里最本质的一条。

**核心操作：**
动笔之前，先问：**这篇文章的读者是谁？他在做什么决策？他会按什么顺序提问？**

然后，用读者的提问顺序来组织文章结构——不是"我有哪些内容要说"，而是"他会依次想到哪些问题"。

```
读者是谁（角色/场景）
    ↓ 他在这个场景下，第一个自然会问什么？
文章第一大点
    ↓ 这个问题回答了，他接下来会问什么？
文章第二大点
    ↓ 以此类推
文章第N大点
```

**为什么这个方法比"按内容逻辑组织"更有力：**
按内容逻辑组织，文章是"我的东西的排列"；按读者决策链组织，文章是"帮读者走完一段思考"。前者是输出，后者是对话。读者感受到的是：这篇文章好像在回答我脑子里的问题。

**实例：**
《AI时代产品机会的底层逻辑》的读者是"战略者"，他在考虑一个产品机会时，会依次问：
1. 要解决什么问题？（→ 文章第一大点）
2. 有哪些设计原则？（→ 文章第二大点）
3. 如何做产品规划？（→ 文章第三大点）

文章的三大点不是三段内容的堆叠，而是战略者在连贯思考中自然流出的三个问题的顺序回答。

---

### 1.2 苏格拉底式建坐标系：先定义范畴，再填内容

在展开任何论证之前，先建立一个让所有内容都能归位的坐标系。

**操作：**
- 找到这个主题最底层的分类结构（不变量，任何时代都成立）
- 先向读者展示这个结构，让他接受"这是分析这个问题的正确框架"
- 然后再把具体内容填进这个框架

**为什么叫"苏格拉底式"：**
苏格拉底在讨论任何问题前，总是先确定定义——"我们说的勇气是什么意思？"。这不是废话，而是在给讨论建立坐标。坐标一旦建立，后面所有的推导都有了落脚点；坐标不建立，内容越多越乱。

**实例：**
分析"AI时代有哪些产品机会"时，先建立坐标系：
> 所有产品问题，永远只来自三类来源：人的问题 / 工具的问题 / 人与工具协作的问题。

这个三分法是任何时代都成立的不变量，先让读者接受它，后面的所有内容都自然归位。

**与"不变量×变量"结构的关系：**
建坐标系是找不变量；填内容是代入变量。两者是同一个思维习惯的不同阶段。

---

### 1.3 详略得当：新旧内容的配比

文章不是越新越好，也不是越长越好。好文章有节奏——有读者已知的内容作为地基，有新洞察作为跃升。

**原则：**
- **旧内容（读者已知的）**：简要带过，作用是"建立共识"和"铺垫"，让读者站稳脚跟，为接受新东西做准备
- **新内容（真正的洞察）**：重点展开，这是文章的真实贡献，值得花笔墨说清楚

**操作：**
写完初稿后，问自己：
> 这篇文章里，哪些是真正的新洞察？哪些是大家本来就知道的？

新洞察的篇幅要多，旧内容的篇幅要省。两者比例失调——全是新的（读者难以接受），或全是旧的（没有价值）——都会让文章失去力量。

**实例：**
《AI时代产品机会的底层逻辑》的第六章有五条判断原则。前四条（马斯洛定位、AI放大、需求质量、Taste）是通用的、普遍认可的，简要列表带过；第五条（研发壁垒）是 AI 时代特有的新洞察，值得展开含表格、飞轮、过程资产三个层次。

---

### 1.4 叙事位置：同一内容放在哪里，效果截然不同

好的内容放错了位置，力量会大打折扣。

**核心判断：**
一个内容的位置应该由它的**认知功能**决定：

| 功能 | 最佳位置 |
|---|---|
| 建立坐标系 / 确立前提 | 最前面，在展开之前 |
| 展示规律的普遍性 | 在具体案例之前（先看规律，再看案例是规律的实例） |
| 最重要的结论 | 段落的末尾，或结语的最前面 |
| 反例 / 限制条件 | 结论之后，作为补充而非削弱 |
| 历史比较 / 跨时代表格 | 在"AI是这个规律的最新案例"之前（让读者先接受规律，再接受 AI 的特殊性） |

**实例：**
在《AI时代产品机会的底层逻辑》的第二章，历史跃迁表格应放在 AI 详细介绍之前——让读者先看到"每次跃迁都遵循同一规律"，然后再读 AI 的内容时，会自然理解"AI 只是这个规律的最新一次代入"，而不是把 AI 当成一个孤立的特殊事件。

---

### 1.5 先找结构，再填内容

写文章的瓶颈很少是"没东西写"，而是"没想清楚逻辑"。动笔之前，先把核心逻辑链条写成一行话：

> **这篇文章的论点是什么 → 为什么 → 有什么证据 → 推导出什么结论**

如果这一行话写不出来，说明还没想清楚，不要开始写。

---

### 1.6 所有核心观点要能独立成段

好文章的每一段都有一个独立的命题，段落的第一句或最后一句是这个命题的最精炼表达。读者只看每段的首尾句，就能读懂文章的骨架。

---

### 1.7 结语要有独立价值

结语不是"总结一遍"，而是把全文的核心命题压缩成一句可以独立传播的话。这句话要能离开文章单独存在，并且让读者读完之后还想把文章发给别人。

---

## 二、分析结构：不变量 × 变量 → 结论

**适用场景：** 分析时代趋势、技术变迁、行业演化等"变化"类主题

**核心逻辑：**
```
先找不变量（跨时代不变的规律本身）
    ×
当前的变量（新技术/新条件发生了什么变化）
    ↓
推导结论（不是猜测，是不变量的必然导出）
```

**为什么这个结构有力：**
不变量是跨时间的锚点。找到一个真正的不变量，就拥有了一个在任何时间点都有效的判断工具——AI 再怎么发展，不变量不会失效，基于它推导的结论同样如此。

**写作时的呈现方式——跨时代对比表格：**

把"不变量 × 变量 → 结论"这个结构，在多个时代里同时铺开，让读者一眼看出规律，而不是被单个时代的细节淹没。

**操作要点：**
1. **在表格前显式说明不变量是什么**，让读者知道"这一列在演示同一个规律的不同时代表现"
2. **最后一行留白或留问号**——规律不会在当前时代停止，把未来的推导留给读者，激发主动思考
3. **表格配短段落**：表格让规律自己说话，段落解释为什么这个不变量是可靠的

**案例一：马斯洛五层 × 生产力工具演进**

不变量：马斯洛五层需求的向上涌动规律（底层被满足后，注意力向更高层流动）

| 跃迁 | 哪类劳动成本↓ | 哪层需求满足成本↓ | 释放出的新需求 |
|---|---|---|---|
| 农业革命 | 食物生产成本 | 第一层 | 人有余力建立定居社会，第二层涌现 |
| 工业革命 | 体力劳动成本 | 第一、二层 | 大规模脑力劳动出现，第三层扩展 |
| 互联网 | 信息传递成本 | 第三层连接成本 | 社交、内容、知识产业爆发，第四层扩展 |
| **AI/智能体** | **脑力劳动成本** | **第四、五层执行成本** | **正在展开** |

**案例二：护城河迁移规律 × 技术平等化能力**

不变量：某类能力被平等化后，护城河自动迁移到下一个更难被复制的层次

| 驱动力 | 被平等化的能力 | 消失的护城河 | 涌现的新护城河 |
|---|---|---|---|
| 印刷术 | 信息记录与传播 | 口耳相传的信息垄断 | **知识本身** |
| 工业革命 | 体力生产 | 物理力量与土地 | **认知技能** |
| 互联网 | 知识获取成本 | 知识本身的稀缺性 | **领域 knowhow** |
| **AI/智能体** | **认知执行（脑力劳动）** | **领域 knowhow** | **研发能力** |
| 未来更强 AI | ？ | ？ | ？ |

## 三、思维升维的写作路径

**适用场景：** 当一个小点可以推演到更大的命题

**过程：**
```
从具体小点出发（不要直接写大命题）
    ↓ 追问：背后的共性是什么？
升一个维度
    ↓ 追问：这里还有没有更本质的规律？
找到底层不变量
    ↓
以此为锚，重新组织文章
```

**写作技巧：** 文章的开头从具体的小点或现象切入，结尾在更高维度收束——读者跟着走完全程，会感受到一种"升维"的满足感，而不是被灌输大道理。

## 四、表格的使用原则

表格是最高效的"让规律可见"工具，但有使用条件：

**适合用表格的情况：**
- 同一个结构在多个维度/时代/对象上重复出现
- 需要对比两种范式/旧与新/变与不变
- 有多个并列的判断标准或维度

**不适合用表格的情况：**
- 只有一两行，信息量太少（用列表更清晰）
- 逻辑是线性推导的（用流程图或段落）
- 内容本身需要大量文字解释（表格会变得臃肿）

**表格设计要点：**
- 列名要是判断标准，不是描述标签
- 最重要的列放在最右侧（读者会自然扫视到最后一列寻求结论）
- 加粗当前时代或最重要的行，帮助读者定位

## 五、ASCII 结构图：让空间关系可见

表格擅长对比属性，ASCII 图擅长表达**空间与结构关系**——当两个概念的位置关系本身就是信息时，ASCII 图比文字更直接。

### 5.1 什么时候用 ASCII 图（而不是表格或文字）

| 适合用 ASCII 图 | 适合用表格 | 适合用文字 |
|---|---|---|
| 东西之间有位置关系（上/下/内/外/之间） | 对比多个对象的属性 | 线性推导过程 |
| 分层结构（层与层之间的关系本身有意义） | 多维度对比 | 描述、解释 |
| 二维矩阵（时间轴×并列维度） | 枚举条目 | 叙事 |
| 嵌套包含关系 | | |

**核心判断**：如果你的脑子里有一张"草图"——东西摆在哪里、谁包含谁、谁在谁旁边——就用 ASCII 图把这张草图画出来。

---

### 5.2 三种常用 ASCII 图类型

**类型一：空间关系盒（Box）**

适用：多个维度/类别之间有空间上的从属或并列关系，某一类在其他类"之间"的位置本身有含义。

```
┌──────────────────────────────────────────────────┐
│                                                  │
│   维度 A                        维度 B           │
│   （A 的说明）                  （B 的说明）      │
│                                                  │
│               维度 C                             │
│               （C 处于 A 和 B 之间/交叉处）       │
│                                                  │
└──────────────────────────────────────────────────┘
```

实例：产品机会三维度——人的问题（A）、工具的问题（B）、人与工具协作的问题（C 处于二者之间）

---

**类型二：分层堆叠图（Stack）**

适用：有明确的上下层次关系，每层的内容不同，层与层之间有交接关系。

```
┌─────────────────────────────────────────────────┐
│  上层名称                                       │
│  关键词 · 关键词 · 关键词                        │
├─────────────────────────────────────────────────┤
│  中间层名称（两层之间的桥接）                   │
│  关键词 ↔ 关键词 · 关键词                       │
├─────────────────────────────────────────────────┤
│  下层名称                                       │
│  关键词 · 关键词 · 关键词                        │
└─────────────────────────────────────────────────┘
```

实例：Human-Readable + Agent-Operable 的三层设计——人层 / 耦合层 / 智能体层

---

**类型三：二维矩阵图（Matrix）**

适用：两个维度同时变化，且它们的交叉位置有意义。常见的是"时间轴 × 并列类别"。

```
时间轴 ──────────────────────────────────→
         现在          近期          中期

A ─────────────────────────────────────→ A'
B ←── 横向强耦合 ──→ 合并为 AB
C               ↓纵向时序耦合↓
                A的出口→A'的入口
```

实例：产品路线图——横轴是时间，纵轴是并列产品，横向耦合和纵向耦合用箭头表示

---

**类型四：循环飞轮图（Flywheel）**

适用：描述一个自增强的正向循环，各节点之间有因果依赖关系，循环一旦启动会自我加速。

四节点方形布局——四个角各放一个状态节点，顺时针箭头表示流向，箭头上标注流动的内容，节点内两行：节点名称 + 节点的核心性质。

```
  ┌───────────────────┐ ──[箭头标注：流动内容]──→ ┌───────────────────┐
  │    节点 A          │                            │    节点 B          │
  │  A 的核心性质      │                            │  B 的核心性质      │
  └───────────────────┘                            └───────────────────┘
          ↑                                                 │
    [左侧箭头标注]                                   [右侧箭头标注]
          │                                                 ↓
  ┌───────────────────┐ ←──[箭头标注：流动内容]──── ┌───────────────────┐
  │    节点 D          │                            │    节点 C          │
  │  D 的核心性质      │                            │  C 的核心性质      │
  └───────────────────┘                            └───────────────────┘
```

实例：研发-产品正向飞轮（产品被使用 → 数据积累 → 研发能力提升 → 产品效果更好 → 更多使用 → …）

---

1. **位置即信息**：ASCII 图里什么在中间、什么在上方、什么被包含——这些位置本身就是在传达含义，不需要文字解释
2. **简洁优先**：不要在图里塞太多文字。关键词足够，解释放在图的前后
3. **图配短段落**：图让结构可见，段落解释为什么这个结构是对的——二者互补
4. **中文字符对齐**：中文字符宽度是英文的两倍，要用全角空格或手动调整对齐

### 6.1 两个层次分开管理

写作的本质不是从零生产内容，而是**把已有的知识原子，按特定的逻辑重新排列**。

```
底层：知识原子层（只增不改）
└── 碎片 → 小闭环 → 原则 → 细节
└── 每次讨论和思考都在积累知识原子
└── 这些原子永远不会被推翻，只会增加

表层：组织逻辑层（每次重新排列）
└── 不同的文章目的 → 不同的排列方式
└── 同样的知识原子，可以按读者的问题排，按时间线排，按逻辑链排
└── 文章是知识原子的一次特定组合，不是从零生产
```

**这与认知成长的方式是同构的：**
碎片 → 小闭环 → 找背后逻辑 → 用不同逻辑展开。

---

### 6.2 写作的正确流程

**第一步：把所有相关知识原子捞出来（往多了列）**

当提出一个写作任务时，先从已有的思维碎片、框架文档、讨论记录中，把所有相关的知识点都列在桌面上。不遗漏、不精选、不提前删减。

**第二步：确认知识原子的完整性**

确认"这些都是我已有的、确认过的知识"，有遗漏的补充进来。

**第三步：讨论这次的排列逻辑**

根据文章目的和读者，决定用什么逻辑组织：
- 读者是谁？他会按什么顺序提问？（换位思考原则）
- 这次的坐标系是什么？（苏格拉底式建坐标）
- 哪些是新洞察要展开，哪些是已知要简要带过？（详略原则）

**第四步：写完整版，先多后少**

先按确定的逻辑写出完整版，不提前删减。确保所有知识原子都有位置。

**第五步：讨论删减**

在完整版的基础上，讨论哪些可以省略或合并，而不是一开始就写精简版。

---

### 6.3 对 AI 协作的具体要求

当提出写作任务时，AI 的默认流程应该是：

1. **先搜索已有文档**（思维碎片、框架文档等），把所有相关知识列出
2. **等确认后**再讨论排列逻辑
3. **先写完整版**，然后根据反馈删减

不应该直接按某个逻辑写一个看起来"干净"的版本——那样会遗漏已经积累的知识原子。

---

### 6.4 AI 能做的事情必须自己做，不能甩给用户

**核心原则：凡是 AI 可以直接执行的操作，必须自己执行，不得要求用户代劳。**

典型的应该自己做但曾经甩给用户的事：
- 运行 `ipconfig` 查 IP 地址 → 直接用 Shell 工具运行
- 启动服务器/代理/脚本 → 直接用 Shell 工具启动
- 检查文件是否存在、查看目录结构 → 直接用 Glob/Read 工具
- 修改配置文件 → 直接用 StrReplace/Write 工具
- 运行验证脚本确认功能 → 直接运行，然后汇报结果

**只有以下情况才需要用户操作：**
- 需要用户输入账号密码/敏感信息
- 需要用户在手机/物理设备上操作（如手机微信扫码、手机设置代理）
- 需要用户做出判断或决策（如"你想要A还是B？"）
- 需要用户在外部网站/平台操作（如微信公众号后台发布）

**出现"你需要做X"时，先问自己**：我能直接做X吗？如果能，直接做，不要问用户。

---

*本节对应认知模型：碎片 → 小闭环 → 背后逻辑 → 多种排列方式*

## 七、反面案例：常见写作问题与原因

> 此节记录在写作过程中发现的具体问题，附原因分析，供写作时对照检查。

---

### 问题一：坐标系部分混入了变量层的内容

**错误写法：**
在建立"普适坐标系"（任何时代都成立的框架）时，举的例子全是当前变量（AI）的具体问题。比如工具问题里列"幻觉问题、跨会话记忆缺失"，协作问题里列"AI不随人成长"——这些是 AI 时代特有的，不是普适的。

**为什么错：**
坐标系的作用是"先让读者接受一个任何时代都成立的框架"，再把当前变量代入其中。如果坐标系里混入了变量层的内容，读者就无法感知到这个框架的普遍性——它看起来只是在描述 AI，而不是在建立一个跨时代的分析工具。

**正确做法：**
坐标系部分只说普适维度（工具问题永远有：效率、准确性、可靠性、成本……；协作问题永远有：意图传达、信任、控制权、上下文……）。AI 的具体问题留到"引入变量"部分再说。

---

### 问题二：替读者预设反驳，显得不自信

**错误写法：**
> "这不是未来的预测，而是一个正在发生的结果。"

**为什么错：**
这句话是在为自己的结论辩护——它预设了读者会质疑"这只是预测"，然后先行防守。好的结论不解释自己为什么可信，直接陈述即可。加了这句话，反而让读者感觉作者对自己的判断不够笃定。

**类似的问题句式：**
- "这不是X，而是Y"（先设假想敌再反驳）
- "很多人可能会认为……但实际上……"（不必要的铺垫）
- "这一点很重要"（让读者评判重不重要，不要自己标注）

**正确做法：**
直接陈述结论，不加防守性修饰。读者自会判断。

---

### 问题三：小节标题是过程描述，不是结论

**错误写法：**
> 小节标题："AI 的技术本质：两个循环构成的闭环执行系统"

**为什么错：**
这个标题描述的是"AI 是什么"——是过程和机制，不是结论。读者看完这一节应该得到的结论是"所以脑力劳动边际成本趋于零"，标题应该是这个结论，而不是正文要讲述的机制。

**正确做法：**
标题 = 这一节读完后读者应该接受的结论。机制和推导放在正文里。

---

### 问题四：元评论式的过渡句——"AI 腔"的典型表现

**错误写法：**
> "核心推论只有一句话："
> "这张表说明一件事："
> "有以下几点值得关注："

**为什么错：**
这类句子是在告诉读者"我接下来要说一个重要的东西"，但它本身没有任何信息量。这是 AI 生成文本的典型模式——用元评论填充过渡，掩盖直接表达的缺失。

真正有力的表达是直接说出那个"核心推论"或"重要的东西"，不需要先宣告它的重要性。

**正确做法：**
- ❌ "核心推论只有一句话：智能体可以承接大量脑力劳动……"
- ✅ "所以，智能体可以承接大量脑力劳动……"
- ❌ "这张表说明一件事：不需要猜测 AI……"
- ✅ 直接开始下一段："不需要猜测 AI 会做什么……"

---

### 问题五：表格后的过渡句重复表格内容

**错误写法：**
> 表格之后写："这张表说明一件事：不需要预测 AI 会做什么，只需要知道脑力劳动成本趋零这件事成立，剩下的是不变量的代入题。"

**为什么错：**
如果表格设计得好，它自己会说话——读者看完表格已经能感受到规律了。用一句话"总结表格说明了什么"，是在重复信息，且会让表格显得不够有力。

**正确做法：**
表格之后直接接推论或下一步的内容，不要先解释表格的含义。让表格自己说话，然后往前走。

### 问题六：标题写的是话题/方法，而非结论

**错误写法：**
> `产品机会的来源：三类永恒坐标 × AI 带来的成本变化`

**为什么错：**
这个标题说的是"我用什么框架来分析这个问题"——是分析的入口，不是分析的结论。读者看完这章之后应该接受的是一个判断，不是一个方法名称。

**类似的错误模式：**
- "X 的来源：A × B"（描述分析框架）
- "X 的结构：A、B、C"（描述内容组成）
- "X 的维度：……"（描述分类方式）

这些都是在说"这章用什么视角"，不是在说"这章得出了什么结论"。

**正确做法：**
问自己：读者读完这一章，应该接受/相信什么？那个判断就是标题。

---

### 问题七：标题写的是机制，而非本质

**错误写法：**
> `为人与智能体同时设计两层界面`

**为什么错：**
"两层界面设计"是实现机制，回答的是"如何做"。但标题应该是本质判断——为什么要这样做，这意味着什么，产品的本质定位是什么。

在这个例子里，两层设计背后更本质的结论是：**产品的本质定位变了——它是人与其数字员工协作的介质**。两层设计是这个本质的实现方式，不是结论本身。

**区分方法：**

| 类型 | 回答的问题 | 适合放在 |
|---|---|---|
| 机制/方法 | 如何做 | 正文 |
| 本质判断 | 这意味着什么 / 本质是什么 | 标题 |

**操作原则：** 如果标题里有"设计/使用/实现/建立"等动词，往往是在描述机制。把它转化成"这个机制揭示了什么本质"，那才是标题。

---

### 问题八：结论蕴含的信息量不够——用定性替代了原则

**错误写法：**
> 章节标题：`产品的本质变了：人与其数字员工协作的介质`

**为什么错：**
"介质"只是一个**定性**——它告诉读者产品是什么，但不告诉读者应该怎么做。定性的信息量低于原则，因为原则从定性推导出来，且包含了行动指向。

真正的结论是从"介质"这个定性推导出的设计原则：`Human-Readable + Agent-Operable`——这个原则蕴含了两类用户、两层设计、碳硅差异……信息量远高于"介质"这个标签。

**逻辑链：**
```
事实（产品有两类用户）
    ↓
定性（产品是人与数字员工的协作介质）——推导路径上的一站，不是终点
    ↓
原则（Human-Readable + Agent-Operable）——才是章节结论
    ↓
展开（两类用户输入器官不同，所以要双层设计）
```

**判断标准：** 在一个定性和一个从定性推出的原则之间，原则蕴含更多信息，应该作为结论。

---

### 问题九：标题描述现象，而非揭示战略含义

**错误写法：**
> `脑力成本趋零，产品机会从执行层转向人性深处`

**为什么错：**
这在描述一个**现象**，让读者理解"发生了什么"，但没有点出战略含义——"所以你应该做什么，或者不该做什么"。

战略者需要的结论是行动指向。真正的战略含义是：**过去的很多问题正在消失，继续解决它们是把错的做对**——这比"脑力成本趋零"这个事实含义更深、指向更明确。

**对比：**
- ❌ `脑力成本趋零，产品机会从执行层转向人性深处`（描述现象）
- ✅ `旧瓶颈正在消失，把它们做好是把错的做对`（揭示战略含义）

前者是观察，后者是判断。

**操作原则：** 问自己：读者看完这一章，应该**做什么**或**停止做什么**？那个答案才是标题。

---

### 问题十一：[待核实]标注不是"保留未核实引用"的合法方式

**错误做法：**

把无法核实的引用保留在文档中，用 `[待核实：具体作者、期刊及DOI]` 标注，认为这样就"处理了"引用问题。

**为什么错：**

[待核实] 是**草稿内部工作标记**，不是一种引用状态。一份含有待核实引用的文档，等于在读者面前陈列了一个可能不存在的信息来源。标注"待核实"并不能降低引用虚假来源的风险。

**硬性规则：**

```
引用进入正式文档（进入 HTML 发布流程）之前，必须满足以下任一条件：
1. 已找到真实发表来源（有 DOI 或可访问的期刊页面）
2. 引用已从文本和参考文献列表中完全删除

[待核实] 标注只允许在主动核实过程中短暂存在，不允许随文档流转至发布阶段。
```

**正确操作：**

发现 [待核实] 引用时，按顺序执行：
1. 搜索核实（WebSearch + WebFetch）
2. 找到真实来源 → 补全引用信息，删除 [待核实] 标注
3. 找不到真实来源 → 删除该引用，同时检查正文中依赖此引用的声明是否需要调整（降低断言强度或删除）

---

### 问题十二：图片提示词太短导致结果不一致，且缺少多视角选项

**错误做法：**

写一条简短的提示词（如"三节点关系图，DMN CEN SN，白色背景"），直接调用一次图片生成 API，嵌入文档。

**为什么错：**

1. **简短提示词无法保证一致性**：不同调用之间，AI 图像生成模型的自由发挥空间会导致结果差异显著。一份用于发布的信息图，需要在任何时候重新生成都能得到基本一致的结果。
2. **单一视角剥夺了选择权**：不同视角的图会传达不同的重点，AI 自己决定用哪个等于替用户做了一个视觉信息架构决策。

**硬性规则：**

```
每个图片位置，必须：
① 提供至少 3 个不同视角的提示词（不是 3 次调用同一个提示词）
② 每条提示词必须包含以下所有元素：
   - 整体布局（几个元素、怎么排列）
   - 每个元素的精确文字内容（复制正文中的确切词汇）
   - 每个元素的颜色（具体色值或明确颜色名）
   - 连接线/箭头的样式和标注文字
   - 特殊高亮或边框（加粗/颜色框）
   - 整体风格词（简洁商务/学术图表/信息图）
③ 展示 3 个视角选项给用户选择，不自行决定
```

**提示词质量判断标准：**

拿着你写的提示词，问自己：如果把这个提示词给另一个人，他能在没有任何其他参考的情况下，画出和你预想一模一样的图吗？如果不能，提示词还不够详细。

**3种常用视角类型（至少选3个不同的）：**

| 视角 | 适合传达的内容 |
|------|------------|
| 结构视角 | 层级/模块/包含关系 |
| 流程视角 | 步骤/时序/因果链 |
| 对比视角 | 两种状态/方案/结果的对比 |
| 关系视角 | 多节点之间的相互影响 |
| 量化视角 | 数据/比例/大小差异（柱状图/饼图） |

---

### 问题十：结语没有收束各章节建立的具体条件——用通用结论替代了逐一闭环

**错误写法：**

文章前几章每章各自建立了一个"缺失条件"（如：反思需要三网络协同、碎片感需要侧路激活、发散需要连续谱保护、学习需要即时反馈），但结语只写了通用的"AI 第一次有机会……低摩擦捕捉碎片……提供即时反馈"，没有把"每个缺失条件"和"AI 的对应补充"显式配对。

**为什么错：**

读者到达结语时问的是：**"前面说的那四个具体条件，AI 具体怎么补上？"** 如果结语只是通用描述，读者会感到"知道 AI 能帮，但不清楚帮的是什么"。

这本质上是：各章节建立了具体的「机制缺口」，结语却没有对每个缺口逐一接住。

**正确做法：**

结语必须把每章建立的「缺失条件」和「AI 的对应补充」显式配对，让读者看到每个章节逻辑的完整闭合：

```
章节建立的缺失条件 → AI 具体如何补上
反思：三网络协同需要外部脚手架 → AI 提供结构化拆解框架，引导三网络协同触发
碎片感：需要零摩擦捕捉 + 跨时间关联 → AI 随时保存 + 跨天主动关联
发散：需要连续谱保护 + 有序整合步骤 → AI 引导不跳步的整合过程
学习：需要即时的针对性反馈 → AI 在每次检索后立即给出错误修正
```

**识别这个问题的方法：**

读完结语，用每章的核心问题检验："这一章说大脑需要 X，结语里有没有说 AI 怎么提供 X？" 如果任何一章的 X 在结语里找不到对应，就是没有闭环。
